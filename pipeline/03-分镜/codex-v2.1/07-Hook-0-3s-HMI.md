# 模块 I · Hook 0-3s 子拆（Hook_Micro）

## 1. 模块定位

本模块（HMI-XX 系列）专门解决买量视频最致命的【前3秒留人】问题，定义了起手式的抛出方式、视觉锚点映射及切刀密度。在 codex-v1.0 中，我们主要通过 STR（结构语法）和 PACE（全局节奏）来控制整体宏观走向，但这导致前 3 秒的微操过于依赖剪辑师的个人体感，未能形成毫秒级的数据约束。v2.0 引入本模块，正是为了通过严格的“认知缺口+视觉锁定”公式，在观众产生划走冲动前，强制将其注意力拽入视频语境。

---

## 2. 核心规则

### HMI-01：疑问先行制造认知缺口（hook_question_first）

**一句话核心**：首句口播必须优先使用疑问句，切忌平铺直叙。
**数据支撑**：76 条样本中，抛出方式以疑问句（Question）为主，占比高达 **52.6% (40/76)**。
**执行方法**：
```json
{
  "hook_info_delivery": "Question"
}
```
**可读约束语句**：Hook 阶段的文案起手式应设计为疑问句，直接抛出玩家痛点或反直觉现象，人为制造“认知缺口（Desire Gap）”，勾起观众寻找答案的本能。
**反例**：
- ❌ “这款游戏今天终于上线了，送很多福利。”（平庸陈述，毫无波澜）
- ❌ “我们的画质是全网最好的。”（王婆卖瓜，观众秒划）

### HMI-02：视觉锚点强绑定（visual_anchor_sync）

**一句话核心**：口播抛出核心名词的瞬间，画面必须有对应的具象化视觉落点。
**数据支撑**：76 条样本中，视觉锚点命中（`visual_anchor_hit`）比例高达 **80.3% (61/76)**。
**执行方法**：
```json
{
  "hook_visual_anchor": true,
  "anchor_sync_latency_ms": 0
}
```
**可读约束语句**：前 3 秒内，画面中必须存在一个强有力且与口播核心名词绝对对应的视觉焦点。观众听到什么，眼睛就必须立刻看到什么，以此快速锁定受众目光。
**反例**：
- ❌ 口播问“历史上真实的白起长什么样？”，画面却是两军交战的远景大混战（丢失视觉锚点）。
- ❌ 口播强调“这把狙击枪的威力”，画面却是人物在跑图。

### HMI-03：极度克制的首 3 秒切刀（hook_shot_limit）

**一句话核心**：开局 3 秒内禁止碎剪，给观众留出识别画面的时间。
**数据支撑**：0-3 秒内的镜头切换次数 `hook_shots_in_3s` 稳定在 **[1, 3] 区间，中位数为 2**。
**执行方法**：
```json
{
  "hook_shots_in_3s_min": 1,
  "hook_shots_in_3s_max": 3,
  "hook_shots_target": 2
}
```
**可读约束语句**：为了确保观众能清晰捕捉到 HMI-02 要求的视觉锚点，前 3 秒内的镜头切换（切刀）必须严格控制在 1 到 3 次之间，默认推荐切 2 刀。
**反例**：
- ❌ 前 3 秒配合动感 BGM 爆切了 6 刀（过度追求视觉冲击，导致严重的光污染，观众无法识别核心信息）。

## 3. 与其他模块的协同

本模块作为 0-3s 的微雕规范，需与 v1.0 的多个底层模块紧密咬合：
- **与 EMO（情绪燃料）配合**：HMI-01 的“疑问句”抛出，完美承载 EMO 模块中的 `Curiosity`（好奇）情绪，两者叠加放大悬念感。
- **与 SUB（字幕色彩）配合**：针对 HMI-02 中的“视觉锚点”对应的口播关键词，强制调用 SUB-01 规则中的“红色（#FF0000）警示色/冲击色”进行字幕高亮。
- **与 PACE（节奏）配合**：HMI-03 的切刀限制实际上是接管了前 3 秒的 PACE 权限，明确禁止开局盲目提速快剪。
- **与 BGM（音轨）配合**：Hook 阶段的 BGM 必须遵循“从静音渐入悬疑/战鼓音”的规则，为 HMI-01 的疑问句留出清晰的人声频段。

---

## 4. 快速 checklist

- ✅ **句式检查**：前 3 秒的口播第一句，是否是抛出悬念的疑问句？
- ✅ **锚点检查**：口播提到的核心名词，画面中央是否出现了清晰、具象的视觉焦点？
- ✅ **节奏检查**：时间轴 0-3s 区域内，镜头切刀数是否 ≤ 3 刀？
- ❌ **避坑**：坚决杜绝“画面与口播各说各的”，引发认知负荷超载。
- ❌ **避坑**：坚决杜绝开局“高频快剪闪瞎眼”，视觉锚点还没看清镜头就切走了。
